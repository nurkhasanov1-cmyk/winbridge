## Context

The non-native agent shell can receive authorization lifecycle messages such as decisions, state updates, controls, permission revocations, and development audit events. Today those messages are local event/log metadata only unless generated by the host workflow, but future UI/native consumers will rely on this surface to represent remote workflow state.

The relay already enforces role and sender boundaries for registered peers. The managed runtime still needs a local defense-in-depth check for tests, tools, or unexpected relay-like endpoints that inject same-session envelopes directly.

## Goals / Non-Goals

**Goals:**

- Ignore decoded inbound `session-authorization-decision` messages whose `hostPeerId` equals the local runtime `peerId`.
- Ignore decoded inbound `session-authorization-state`, `session-control`, `permission-revoked`, and `audit-event` messages whose `actorPeerId` equals the local runtime `peerId`.
- Run the check before local `received` protocol event emission or received workflow summary logging.
- Keep ignored-message diagnostics secret-safe by exposing only redacted summary metadata such as byte length.
- Preserve normal remote-to-local workflow messages from distinct peers.

**Non-Goals:**

- No protocol schema, relay behavior, production identity, token lifecycle, capture, input, clipboard, file transfer, installer, service, startup, privilege, or native Windows behavior changes.
- No change to outbound host workflow simulation.

## Decisions

- Add a local self-authority guard after protocol decoding and session matching, before received-event emission.
  - Rationale: authority peer ids can be inspected only after decoding, and the guard must run before local workflow metadata is surfaced.
  - Alternative considered: rely only on relay role checks. Rejected because the agent-shell runtime should fail closed against unexpected relay-like sources.
- Reuse the unsafe inbound protocol redaction path.
  - Rationale: ignored authority input should not expose message type, peer ids, authorization ids, audit ids, actions, reasons, tokens, or payload fragments in local raw events/logs.
  - Alternative considered: emit a specific self-authority mismatch diagnostic. Rejected because generic byte-length metadata is sufficient and safer.

## Risks / Trade-offs

- Misconfigured peers that reuse the local peer id for authority metadata will be ignored. -> Mitigation: distinct peer ids are required for meaningful remote authority, and fail-closed behavior preserves consent boundaries.
- Generic ignored-message diagnostics reduce debugging detail. -> Mitigation: byte length remains available without exposing workflow metadata or secrets.
