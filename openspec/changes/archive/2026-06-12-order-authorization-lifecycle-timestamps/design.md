## Context

`SessionAuthorizationSchema` is the shared gate that future relay, agent, and native adapters will parse before any sensitive remote action check. It already validates permission scope, host visibility, terminal permissions, required lifecycle timestamps, forbidden pre-active/denied timestamps, and each lifecycle timestamp's placement within the record's creation-to-update window.

The remaining gap is relative lifecycle ordering. A record can currently keep timestamps inside the window while still implying impossible history, such as activation before approval or termination before activation. That inconsistency should be rejected at schema parse time rather than left for downstream action checks or audit consumers to interpret.

## Goals / Non-Goals

**Goals:**

- Reject authorization records whose lifecycle timestamps contradict the consent-first transition order.
- Keep the validation local to `packages/protocol/src/authorization.ts`.
- Preserve existing valid records produced by the state-machine helpers.
- Add tests that prove malformed timestamp order is rejected before action authorization can use the record.

**Non-Goals:**

- No changes to protocol message shapes, relay routing, agent-shell workflow timing, audit persistence format, native capture, input, installer, startup, services, tokens, or privilege behavior.
- No production identity or account model.
- No attempt to model unlimited pause/resume history; the current single `pausedAt`/`resumedAt` fields remain unchanged.

## Decisions

- Add a schema-level timestamp ordering helper instead of transition-specific checks in action authorization. Schema parsing is the earliest common boundary for imported, persisted, or protocol-derived authorization records.
- Compare only optional fields that are present. This preserves pending and approved records that have not reached later lifecycle states.
- Keep partial permission revocation valid for live authorizations when `revokedAt` is after approval/activation. Partial revocation is fail-closed for the revoked permission but can leave the authorization active or paused for remaining permissions.
- Reject `deniedAt`, `terminatedAt`, and `expiredAt` on live `active` or `paused` records. These timestamps represent fail-closed outcomes and must not coexist with a live authorizing status.
- Validate terminal timestamps after their prerequisite lifecycle timestamps. Revocation, termination, and active/paused expiration must not appear to occur before approval or activation when those prerequisite timestamps are present.

## Risks / Trade-offs

- [Risk] Stricter parsing can reject previously accepted malformed fixture data. -> Mitigation: update only focused tests and preserve records generated by the public state-machine helpers.
- [Risk] Single `pausedAt`/`resumedAt` fields cannot represent a full multi-cycle pause history. -> Mitigation: validate the currently represented latest pause/resume order without changing the data model.
- [Risk] Authorization validation changes affect sensitive access decisions. -> Mitigation: run focused protocol tests, full verification, OpenSpec validation, and a focused security review before archive/release.
